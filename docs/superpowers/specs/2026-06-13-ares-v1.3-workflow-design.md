# Ares v1.3 "Flujo" — Diseño

**Fecha:** 2026-06-13 · **Estado:** Aprobado por Marc

## Objetivo

Que Ares, ante una tarea no trivial, siga un flujo disciplinado en fases con
revisión entre ellas —**spec → review → plan → review → implementación
(subagentes) → review de código → resumen → review**— y que el paso de code
review use la skill correcta según el repo/tecnología, dando precedencia a la
skill que ya vive en el propio proyecto (ej. `code-review-discipline` de
dafne-api).

## Decisiones cerradas

| Tema | Decisión |
|---|---|
| Cómo usa Ares la skill de review | **Lee el `SKILL.md` y lo sigue como protocolo** (no depende de que el motor cargue `.claude/skills/` como skill registrada). |
| Descubrimiento (3 niveles) | 1) repo `.claude/skills/*review*/SKILL.md` gana · 2) detección de tecnología → `~/.ares/skills/review-<tech>/SKILL.md` · 3) review genérico incorporado. |
| Qué revisa cada fase | Adaptado a la fase: spec→review de spec, plan→review de plan, código→skill técnica del repo, resumen→review de fidelidad. La skill específica del repo solo entra en la fase de código. |
| Disparo | Automático según el tamaño: trivial (fix/pregunta) → directo; no trivial (feature, varios archivos, dominio nuevo) → flujo completo. |
| Investigación | La fase de spec **siempre** investiga primero (WebSearch + context7): cómo lo resuelven otros, cuál es el consenso/forma correcta. No es opcional. |

## Mecanismo de descubrimiento de la skill de review

`core/review.ts` expone `findReviewSkill(cwd: string): { path: string; source: "repo" | "tech" | "generic"; tech?: string }`:

1. **Repo gana.** Busca en `<cwd>/.claude/skills/` un directorio cuyo nombre
   contenga `review` con un `SKILL.md`. Si existe → devuelve esa ruta
   (`source: "repo"`).
2. **Tecnología como fallback.** Detecta el stack por archivos marcadores:
   - `Cargo.toml` → `rust`
   - `package.json` → `typescript` (o `node`)
   - `pyproject.toml` / `requirements.txt` → `python`
   - `project.godot` → `godot`
   - `go.mod` → `go`
   Busca `~/.ares/skills/review-<tech>/SKILL.md`. Si existe → lo devuelve
   (`source: "tech"`).
3. **Genérico.** Si nada aplica → review genérico incorporado
   (reuse/quality/efficiency sin sesgo de stack), `source: "generic"`.

El helper devuelve la ruta y el contenido; el protocolo de Ares lo lee y sigue
sus instrucciones (incluido el patrón de 3 agentes paralelos cuando la skill lo
define).

## El flujo (protocolo `workflow.md`)

Ares lo activa automáticamente cuando la tarea es no trivial (feature, ≥3
archivos previsibles, dominio nuevo, refactor de calado). Para fix de una línea,
pregunta o cambio trivial: directo, sin flujo.

1. **Investigación + Spec.** Primero investiga (WebSearch + context7: enfoques de
   varios, consenso, forma correcta). Luego escribe la spec. → **review de
   spec**: ¿completa, sin ambigüedades, sin contradicciones, alcance correcto?
2. **Plan.** Descompone en pasos atómicos y verificables. → **review de plan**:
   ¿cubre toda la spec? ¿cada paso tiene su verificación?
3. **Implementación.** Tarea por tarea con subagentes (tool Task), siguiendo los
   protocolos existentes (step-by-step, verification). → **review de código**:
   localiza la skill con `findReviewSkill(cwd)` y la sigue (los 3 agentes
   reuse/quality/efficiency si la skill del repo lo define).
4. **Resumen.** Qué se hizo. → **review de fidelidad**: ¿el resumen refleja la
   verdad? ¿algún test saltado, algo a medias sin declarar, alguna afirmación
   sin verificar?

Cada review que encuentra problemas vuelve a la fase correspondiente antes de
avanzar.

- `tests/review.test.ts` — los 3 niveles de descubrimiento + detección de tech.

## Manejo de errores

- Sin `SKILL.md` en ningún nivel → review genérico, avisando de la fuente usada.
- WebSearch/context7 no disponibles (p. ej. run headless sin MCP) → la fase de
  investigación lo registra como "no verificado" y sigue, no bloquea.
- Repo sin tecnología reconocida → genérico.

## Testing

- Unit (`review.test.ts`): repo con `.claude/skills/x-review/` → source repo;
  repo con `Cargo.toml` y sin skill local → source tech `rust`; repo vacío →
  generic; precedencia repo > tech con ambos presentes.
- Integración: tarea no trivial en un repo de prueba → comprobar que pasa por
  las fases y que el review de código localiza la skill correcta.

## Fuera de alcance (v1.3)

  gana).
- Comando manual `/flujo` (el disparo es automático; trivial de añadir luego).
