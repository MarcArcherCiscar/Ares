# Ares v1.4 "Craft" — Diseño

**Fecha:** 2026-06-13 · **Estado:** Aprobado por Marc

## Objetivo

Acercar a Ares al nivel de un programador de primer nivel reforzando las tres
disciplinas que más correlacionan con ingeniería de élite y que no dependen solo
del modelo: tests que conducen el diseño (TDD), comprensión estructural del
código (ojos reales, no grep) y memoria arquitectónica por proyecto (el
conocimiento que un senior carga de cada repo).

## Decisiones cerradas

| Tema | Decisión |
|---|---|
| TDD — estrictez | Estricto en lógica nueva no trivial (red→green→refactor); bug → test que reproduce primero; flexible (sin forzar) en config, estilos, prototipos visuales, docs. |
| Ojos estructurales — fuente | MCP de CodeGraph, opcional. Repo con `.codegraph/` → se conecta. CodeGraph instalado pero repo sin indexar → **pregunta a Marc** antes de `codegraph init`. No instalado → silencioso, grep/glob como fallback. |
| Memoria arquitectónica — dónde | `<repo>/.ares/NOTES.md`, autoescrita por Ares con tools de archivo normales, cargada al arrancar en ese repo. Viaja con el repo (Marc decide si versiona o ignora). |

## Feature 1 — Disciplina TDD

**`soul/protocols/tdd.md`** (nuevo):
- Lógica nueva no trivial: escribe el test que falla, vélo fallar, implementa lo
  mínimo, vélo pasar, refactoriza. No saltarse el "vélo fallar".
- Bug: primero un test que lo reproduce (rojo), luego el fix (verde).
- No testeable (config, estilos CSS, prototipos de UI, docs, cambios triviales):
  no fuerces tests; di por qué no aplica si no es obvio.
- Encaje: la fase 3 del flujo (`workflow.md`, implementación) referencia este
  protocolo; la doctrina de `soul.md` añade el punto correspondiente.

## Feature 2 — Ojos estructurales (CodeGraph MCP)

**`src/core/codegraph.ts`** (nuevo): `codegraphStatus(cwd): "ready" | "needs-init" | "absent"`
- `ready`: existe `<cwd>/.codegraph/` → Ares conectará el MCP de CodeGraph.
- `needs-init`: el binario `codegraph` está en PATH pero no hay `.codegraph/` →
  Ares debe preguntar a Marc antes de inicializar.
- `absent`: ni `.codegraph/` ni binario → grep/glob, sin mención.

**Conexión del MCP:** cuando el estado es `ready`, `core/agent.ts` añade el
servidor MCP de CodeGraph a `options.mcpServers` (servidor externo stdio). El
comando de lanzamiento exacto se verifica en el plan contra la doc de CodeGraph.

**`soul/protocols/structural-eyes.md`** (nuevo): cuando las tools
`codegraph_*` están disponibles, úsalas para preguntas estructurales (dónde se
define X, quién llama a Y, qué rompe cambiar Z) en vez de grep; grep queda para
búsquedas de texto literal. Si el estado es `needs-init`, ofrece a Marc
inicializar codegraph (una vez por sesión, no insistir).

## Feature 3 — Memoria arquitectónica por proyecto

**`<repo>/.ares/NOTES.md`**: notas de arquitectura del repo (gotchas,
decisiones, campos minados). Ares las escribe con sus tools de archivo normales
(Write/Edit) — no hace falta tool nueva.

**Carga al arrancar:** `core/agent.ts`, junto a la inyección del índice de
memoria global, lee `<cwd>/.ares/NOTES.md` si existe y lo añade al system prompt
bajo una sección "# Notas del proyecto".

**`soul/protocols/project-notes.md`** (nuevo): qué anotar (lo no obvio que costó
descubrir: por qué tal decisión, qué parece roto y no lo está, dónde están los
pies). Qué NO anotar (lo que ya está en el código, el README o git). Cuándo:
cuando descubras algo durante el trabajo que tu yo futuro agradecería saber al
volver al repo.

## Arquitectura

- `src/core/codegraph.ts` — detección de estado (pura, testeable).
- `src/core/agent.ts` — (a) carga `.ares/NOTES.md` al componer el system prompt;
  (b) si codegraph está `ready`, añade su MCP a `mcpServers`.
- `src/core/soul/protocols/{tdd,structural-eyes,project-notes}.md` — protocolos.
- `src/core/soul/soul.md` — la doctrina referencia los tres.
- `tests/codegraph.test.ts` — los tres estados de `codegraphStatus`.

## Manejo de errores

- CodeGraph MCP no arranca o falla → Ares lo registra y sigue con grep/glob, sin
  romper la sesión.
- `.ares/NOTES.md` ilegible → se ignora, no bloquea.
- Repo sin codegraph ni notas → comportamiento actual, sin ruido.

## Testing

- Unit (`codegraph.test.ts`): dir con `.codegraph/` → ready; sin él pero con
  binario simulado en PATH → needs-init; sin nada → absent.
- Integración: en un repo con `.codegraph/`, comprobar que `core/agent.ts`
  incluye el MCP de codegraph en las opciones de `query` (sin llegar a indexar).
- El protocolo TDD se valida con un eval: tarea de feature → comprobar que Ares
  escribe el test antes que el código (orden de los eventos).

## Fuera de alcance (v1.4)

- Indexado automático de codegraph sin preguntar.
- Otros MCP estructurales (LSP) además de codegraph.
- Sincronizar `.ares/NOTES.md` entre repos o con la memoria global.
- Versionar/auto-gitignore de `.ares/` (lo decide Marc por repo).
